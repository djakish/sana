What a search database does.

Before any machinery, the question: what is the job? A search database takes a pile of things and answers questions about them quickly. This chapter sets up the words we will use for the rest of the book.

Things and their fields

Everything starts with the things you want to keep. In Sana each thing is a document: an item with an identifier and some named fields. Our running example is a small library, where each document is a book.

Document

One record, identified by an id. It carries attributes — named values like title or rating — and optionally text and vectors, which we meet soon. A document is the unit you store and the unit a search returns. 

document id 3
  title   "Dune"
  genre   "scifi"
  rating  4.7
  year    1965
  text    "...the spice must flow..."
  vector  [0.91, 0.04, 0.38, ...]

Documents that belong together live in a namespace — a named, self-contained collection. Our five books live in a namespace called books. A namespace is the boundary for everything that follows: its own data, its own indexes, its own catalog. Two namespaces never share files, so one can be rebuilt or deleted without touching another. 

namespace "books"
  1  A Wizard of Earthsea     fantasy  4.5  1968
  2  The Dispossessed         scifi    4.8  1974
  3  Dune                     scifi    4.7  1965
  4  The Left Hand of Darkness scifi   4.6  1969
  5  The Hobbit               fantasy  4.7  1937

Three kinds of question

"Search" is not one thing. People ask three different kinds of question of a collection, and a useful engine answers all three over the same documents. Sana is, in large part, the story of building one engine that does so.

The questionIn wordsExample over the library
Where X — a filterExact, factual conditions on fields."scifi books rated 4.6 or higher."
Mentions Y — full-textDocuments whose words match yours, ranked by relevance."books whose title mentions darkness."
Means Z — vectorDocuments whose meaning is closest to an example."books that feel like Dune."

The first is a matter of yes-or-no facts. The second is about words and relevance. The third is about meaning, and it is the one most people have never seen explained — that is Chapter 5. The payoff of putting all three in one engine is the hybrid query: rank by meaning, then keep only the ones that also pass a filter.

Why searching naively is too slow

Suppose someone asks for "scifi books rated 4.6 or higher." The obvious way to answer is to walk through every document and check each one. With five books that is instant. With fifty million, reading every document on every query is hopeless — most of the work is spent rejecting things.

The fix is an index. The word is borrowed from the back of a textbook: instead of reading every page to find "mitochondria," you look it up in the index and it tells you the three pages that matter. A database index is the same trick — a structure prepared in advance so that a query can jump straight to the few documents that could match, and skip the rest.

Index

A precomputed lookup structure that turns a question into a short list of candidate documents without scanning the whole collection. Sana keeps a different kind of index for each kind of question — one for filters, one for text, one for vectors — and they are the subject of Parts III.

An index is a bargain, not magic: you pay to build and store it up front, and in return every later query is cheap. Much of this book is about building those structures so they stay correct while data keeps changing.

Where the data lives: object storage

A database has to put its bytes somewhere. Most databases manage their own disks and replicate data across machines to survive failure. Sana makes a different choice: it keeps every durable byte in an object store, and nothing else.

Object store

A service that holds named blobs of bytes — Amazon S3 is the famous one; a plain local folder is the simplest. You put bytes under a key (a path-like name) and later get them back. It is not a database: there are no rows, no queries, no transactions across keys. Just keys and bytes.

Why build on something so bare? Because an object store is extraordinarily good at the one thing a database needs most — not losing data. It keeps many redundant copies, it is cheap, it is shared by every machine at once, and it needs no operator to babysit a disk. The cost of the bargain is speed: a single request to S3 takes tens of milliseconds, an eternity next to local memory. Almost every clever thing later in the book exists to hide that latency — caching, batching, and keeping a small buffer of recent writes so the store is not consulted on every read.

There is one more thing the store gives us, and it is the hinge the whole design turns on. A good object store can perform a conditional write: "create this file, but only if it does not already exist," or "replace this file, but only if it still holds the version I last saw." That small guarantee is what lets many machines share one bucket without corrupting each other's work. We unpack it in Chapter 2.

The shape of the journey

The rest of the book climbs from the bytes upward. Each rung adds one idea and assumes the rung below it.

  1. Durability (Ch. 2) — make a write survive a crash using only the object store's conditional writes.
  2. Shape (Ch. 3) — store data in sorted, never-edited files so a key can be found quickly and old files merged away.
  3. Filters and text (Ch. 4) — answer "where X" and "mentions Y" with indexes, over a consistent view that always includes your latest write.
  4. Meaning (Ch. 5–6) — represent documents as vectors and search by closeness, fast enough at scale and fresh as data changes.
  5. Operating (Ch. 7) — run the whole thing as real processes that can start, stop, and share work safely.
What we earned. A vocabulary: document, namespace, the three kinds of question (filter, text, vector), index, and object store. Everything from here on is built out of exactly these pieces.
