02 / storage

Making a write survive.

A database's first promise is simple: once it tells you a write succeeded, that write is never lost — not to a crash, not to two clients at once. This chapter earns that promise using only the handful of operations an object store offers.

The store's small menu

Recall from Chapter 1 that the object store is not a database. It offers about seven operations, and the entire engine is arranged so they are enough. 

ObjectStore
  get(key)                   -> bytes + version
  get_range(key, start..end) -> a slice of the bytes
  put(key, bytes)            -> overwrite, return new version
  put_if_absent(key, bytes)  -> create only if the key is free
  compare_and_set(key, expected, bytes)  -> replace only if unchanged
  list(prefix)               -> keys under a prefix (recovery, not queries)
  delete(key)

Two of these do the heavy lifting. put_if_absent writes a file only if no file already has that name. compare_and_set — usually shortened to CAS — replaces a file only if it still holds the exact bytes you last saw. Both are conditional: the store itself enforces the condition, and it can refuse. That refusal is the feature.

Why "just save the file" fails

Imagine storing the library as one file you overwrite on every change. Two details sink it. First, a crash: if the machine dies halfway through writing the file, you are left with a corrupt mixture of old and new. There is no moment where the change is cleanly "done."

Second, and worse, two writers at once. Both read the current file, both make their edit, both save. Whoever saves last wins, and the other person's change silently vanishes. This is the lost update, the oldest bug in shared storage. 

writer A reads file (rating 4.5)   writer B reads file (rating 4.5)
A sets rating = 4.8                 B sets genre = "fantasy"
A saves  ->  rating 4.8             B saves  ->  genre fantasy, rating 4.5
                                    A's change is gone

Compare-and-set is the cure. A writer says "replace the file, but only if it still holds the version I read." If someone changed it in between, the store rejects the write and the loser simply re-reads and tries again. No edit is ever silently overwritten.

This is optimistic concurrency: don't lock anything up front; just attempt the change and let the store reject you if the world moved. On S3 this check is enforced by the service, so it holds across different machines — several Sana processes can share one bucket safely.

Write once, never edit

CAS handles the few files that must change. Sana's deeper move is to make almost every file immutable — written exactly once and never modified. New data means a new file with a new name, never an edit to an old one.

Immutable object

A file written with put_if_absent and never changed afterward. Because its contents are fixed, its name can be derived from those contents (a content hash), and re-doing an interrupted write just produces the identical file — a harmless no-op. Immutable data can be cached forever and shared freely, because it can never go stale.

Immutability dissolves the crash problem. A half-written immutable file is never referenced by anything, so no reader can ever see it. It is just stray bytes — an orphan — to be swept up later. The only question that remains is: how does a new file become official?

Publish the catalog last

Sana writes all the new data first, and only at the very end flips a single pointer to make it live. That pointer is manifest/current, and the file it names — the manifest — is the namespace's catalog: a list of exactly which immutable files make up the current state. 

1. write immutable data files
     index/g/41/doc/flush-900-….sst
     index/g/41/vector/embedding/ivf-….bin

2. compare_and_set(manifest/current, old -> points to generation 41)

3. new readers follow the pointer and see generation 41
Catalog-last. Because the data exists before the pointer names it, a reader can never follow the catalog to a missing file. And if the final CAS loses a race, the freshly written data files are simply orphans — they never became the catalog. Correctness lives in one pointer, not in scanning a directory.

Durable now, indexed soon

There is a tension. Building all the search indexes for a write takes real work, but the client wants its write acknowledged immediately. Sana resolves this by splitting "saved" from "searchable."

Every write first goes into the WAL — the write-ahead log, an append-only sequence of the raw changes. A write is durable the moment its entry is committed to the WAL. Turning that entry into search indexes happens afterward, in the background. The client never waits for indexing.

WAL (write-ahead log)

An ordered, append-only record of every change, written before anything else. It is the source of truth for what has been accepted. A position in the log is a cursor; the latest committed cursor is the high-water mark of durability. Indexes are just a faster-to-read summary of the log so far.

Committing one WAL entry is a small five-step protocol, and every step is designed so that a crash in the middle is recoverable. Step through it:

The reserve step (the CAS on wal_commit) is the linearization point — the single instant where a write is declared to come before or after every other. Any writer that arrives and finds a half-finished reservation finishes it before taking its own turn, so a crash mid-protocol and two concurrent writers can neither lose nor duplicate a batch.

The files on disk

Putting it together, one namespace is just a tree of keys. Mutable pointers (CAS) are marked; everything else is immutable. 

namespaces/books/
  wal_commit/current          # committed cursor + pending reservation   (CAS)
  wal_staging/0/7-….wal       # staged batch, not yet official           (immutable)
  wal/0/7.wal                 # committed batch                          (immutable)
  manifest/current            # pointer to the live generation           (CAS)
  manifest/g/41.json          # the catalog for generation 41           (immutable)
  index/g/41/…                # the SSTs and vector files it names       (immutable)

What a "version" is

CAS needs to compare versions, so what is one? It is an opaque token. On a local folder Sana computes a hash of the file's bytes; on S3 it uses the store's own ETag. CAS only asks "is this token still equal to the one I saw?" — never what it means. Each mutable file also embeds a strictly increasing number, so a version can never accidentally repeat.

wal_commit/currentCAS replaceThe cursor only ever moves forward.
What we earned. Crash-proof, race-proof writes on top of a plain object store: immutable files that can never be half-seen, one CAS pointer that publishes a whole new state atomically, and a WAL that makes a write durable long before it is indexed. Next we give those files a shape that is fast to read.
