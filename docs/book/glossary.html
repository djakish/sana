appendix / glossary

Every term, in one place.

Every word the book defines, gathered for quick reference. If a chapter moves too fast, look here. The chapter where each idea is introduced is noted in parentheses.

Approximate nearest neighbor (ANN)
Vector search that inspects only the promising regions of space and exact-scores a few survivors, trading a little accuracy for a large speed-up. (Ch. 5)
Attribute
A named value on a document, such as genre or rating. Filters are predicates over attributes. (Ch. 1)
Base and overlay
The base is the indexed data named by the manifest; the overlay is the fresh WAL tail applied on top at query time. Together they give a view that includes your latest write. (Ch. 3)
BM25
The standard full-text relevance formula. It scores a document by how often the query term appears, how rare that term is overall, and how long the document is. (Ch. 4)
Catalog-last publication
Writing all data files before flipping the one pointer that makes them live, so a reader can never follow the catalog to a missing file. (Ch. 2)
Centroid & cluster
A centroid is the center of a group of nearby vectors; the cluster is the group filed under it. IVF searches only the clusters near the query. (Ch. 5)
Compaction
Background merging of many sorted runs into one: keep the newest record per id, drop obsolete tombstones, and rebuild clean indexes so reads stay cheap. (Ch. 3)
Compare-and-set (CAS)
Replace a file only if it still holds the exact version you last read; otherwise the store rejects the write. The basis of safe concurrent updates. (Ch. 2)
Cursor
A position in the WAL. The latest committed cursor marks how far writes are durable; the indexed cursor marks how far indexing has caught up. (Ch. 2–3)
Distance metric
The rule that turns two vectors into one number for how alike they are: L2 (straight-line distance), dot product (alignment), or cosine (angle only). (Ch. 5)
Document
One record: an id plus attributes, optional text, and optional vectors. The unit you store and the unit a search returns. (Ch. 1)
Drain
On shutdown, refusing new traffic while letting in-flight requests finish, so no request is dropped. (Ch. 7)
Embedding
A vector that represents the meaning of something, produced by a model trained so that similar inputs land at nearby points. (Ch. 5)
False positive / false negative
A candidate that does not really match (harmless — it gets rechecked) versus a real match the index forgot (fatal). Sana's indexes allow the first and forbid the second. (Ch. 4)
Generation
A numbered version of a namespace's catalog. Publishing means CAS-advancing manifest/current to a new generation. (Ch. 2)
Hybrid retrieval
Combining different search types — e.g. vector and full-text — over one snapshot and fusing their ranked lists into a single result. (Ch. 4)
Immutable object
A file written once and never changed. It can be content-addressed, cached forever, and shared freely because it can never go stale. (Ch. 2)
Index
A precomputed structure that turns a question into a short candidate list without scanning everything. Sana keeps one per question type. (Ch. 1)
IVF (inverted file)
A vector index that clusters vectors around centroids; a query probes the nearest clusters and scans only those. (Ch. 5)
k-nearest-neighbor (kNN)
Returning the k stored items closest to a query. Exact kNN compares against every item; approximate kNN does not. (Ch. 5)
Lease
A time-limited, fenced claim on a namespace so exactly one indexing worker processes it at a time. (Ch. 7)
Linearization point
The single instant — the CAS reservation on the commit cursor — at which a write is fixed to come before or after every other. (Ch. 2)
Liveness vs. readiness
Liveness: the process is running (else restart it). Readiness: it should get traffic now (else stop routing, but keep it alive). (Ch. 7)
LSM tree (log-structured merge tree)
A stack of immutable sorted runs, newest on top, read newest-first and merged in the background. How Sana stores ever-changing data in never-changing files. (Ch. 3)
Manifest
A namespace's catalog: the immutable file listing exactly which runs and vector files make up the current generation, named by the manifest/current pointer. (Ch. 2)
Namespace
A named, self-contained collection of documents with its own data, indexes, and catalog. (Ch. 1)
Object store
A service that holds named blobs of bytes (S3, or a local folder). Sana keeps every durable byte here and nothing else. (Ch. 1)
Optimistic concurrency
Don't lock up front; attempt a change and let the store reject it if the world moved. Implemented with CAS. (Ch. 2)
Orphan
An immutable file that was written but never became live (a lost race, or an interrupted write). Harmless; reclaimed later. (Ch. 2)
Posting list
For a given value or term, the sorted list of document ids that have it. The building block of attribute and full-text indexes. (Ch. 4)
Probes
How many of the nearest IVF clusters a query scans — the accuracy-versus-speed dial. (Ch. 5)
Quantization
Storing numbers at lower precision to make comparisons cheap. RaBitQ goes down to one bit per dimension. (Ch. 6)
RaBitQ
A quantization scheme that codes each vector's normalized residual as one bit per dimension, with an error bound, so distances can be estimated cheaply and pruned before exact reranking. (Ch. 6)
Recall
The fraction of the true nearest neighbors an approximate search actually returned. The number you push toward 1.0 while reading as little as possible. (Ch. 5)
Recheck (superset rule)
An index may return too many candidates but never too few; each candidate is verified against the live document before it enters the answer. (Ch. 4)
Residual
A vector's offset from its cluster centroid (o − c). Within a cluster, residuals are what distinguish vectors, so RaBitQ codes them. (Ch. 6)
Run
One sorted SST produced by a single flush; runs stack newest-first in the LSM. (Ch. 3)
Snapshot
A single consistent view of a namespace captured at the start of a query, so every part of the answer reflects the same instant. (Ch. 4)
SPFresh (LIRE)
A scheme for keeping a cluster index healthy with local split/merge/reassign repairs instead of global rebuilds. (Ch. 6)
SSTable (SST)
An immutable file of key-value pairs sorted by key, split into blocks with an index, so any key is found in a few small reads. (Ch. 3)
Token
A unit of text for full-text search — roughly a lowercased word. Documents are indexed and queried by their tokens. (Ch. 4)
Tombstone
A marker in a newer run that records a deletion, hiding an older document without editing the run that still holds it. (Ch. 3)
Vector
An ordered list of numbers, read as the coordinates of a point. Closeness between points is the basis of vector search. (Ch. 5)
Version map (liveness)
versions.bin: which version of each id's vector is live. Updates write a new copy and re-point the map; stale copies are ignored, never edited. (Ch. 5)
WAL (write-ahead log)
An ordered, append-only record of every change, written before anything else. The source of truth for what has been accepted; a write is durable once its WAL entry commits. (Ch. 2)
