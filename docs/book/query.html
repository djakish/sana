Answering a question.

With durable, fast-to-read storage in place, we can finally search. A query in Sana is four moves: fix a consistent view, gather a generous list of candidates, recheck each one against the truth, then rank. Those four moves cover filters, text, and — next chapter — vectors.

First, a fixed view

Data is changing all the time. If a query read the catalog now, a run a moment later, and the WAL after that, it could see an inconsistent mix — half of one update, none of another. So a query begins by capturing one snapshot and reading everything from it.

Snapshot

A single consistent view of a namespace, captured at the start of a query: the manifest pointer, the catalog it names, and the committed WAL cursor. From there Sana builds the base-plus-overlay (Chapter 3) and runs the whole query against it, so every part of the answer reflects the same instant — including your most recent write. 

read manifest/current        (which generation is live)
read the manifest body       (which runs and vectors)
read wal_commit/current      (how far writes are durable)
read the WAL overlay         (the fresh tail)
=> one base + overlay, frozen for this query

Gather generously, then recheck

Here is the central idea of the whole query engine. An index does not have to be perfectly up to date, as long as it errs in one safe direction. It may return too many candidates, never too few.

Think about the two ways an index can be wrong. A false positive is a candidate that turns out not to match — annoying, but harmless, because we will check it. A false negative is a real match the index forgot to mention — fatal, because a missing answer is a silently wrong result. Sana designs every index to allow the first and forbid the second.

The naive approach is to trust the index exactly and return what it says. That breaks the moment the index lags a fresh write, or holds a stale value from before an update. Sana's rule instead: the index returns a superset of candidates, and each candidate is rechecked against the live document before it is allowed into the answer. 
attribute index says:   [1, 4, 9, 12]   (a superset — maybe too many)
load those live docs:    1, 4, 9, 12
recheck the filter:     [1, 9]          (4 and 12 didn't really match)
rank / limit:           [9, 1]

Because the live document is always the final judge, the indexes are free to be approximate and cheap. This one rule is what lets delta files, stale values, and not-yet-merged updates all coexist without ever producing a wrong answer. (The single exception is Not: a "everything except" filter needs to know the full universe exactly, so it falls back to a full scan.)

Filters and aggregates

A filter answers the "where X" question. Equality (genre = "scifi") and range (rating ≥ 4.6) predicates are served by the attribute index, which maps each value to a sorted list of the ids that have it — a posting list. And and Or intersect and union those lists.

An aggregate summarizes the matches rather than returning them — a Count, or a Sum of ratings. Aggregates are computed over every matching row, before the returned page is cut down to the requested limit, so "sum of ratings" is honest even when you only ask to see ten rows.

Full-text search, from zero

The "mentions Y" question is different: a search for darkness should find "The Left Hand of Darkness" and rank it by how relevant it is, not just whether the word appears. This is full-text search, and the standard recipe for relevance is BM25.

First the text is broken into tokens — roughly, the lowercased words. For each token Sana keeps a posting list of which documents contain it and how often. Scoring a query then balances three intuitions:

  1. Term frequency. A document that uses the word more is more about it — but with diminishing returns, so ten mentions is not ten times one.
  2. Rarity. A rare word is more telling than a common one. Matching "darkness" says far more than matching "the," so rare terms are weighted up.
  3. Length. A long document has more chances to contain any word, so its matches are discounted to keep things fair against short ones.
BM25

A ranking formula that scores a document for a text query by combining the three intuitions above: how often the term appears, how rare the term is across the collection, and how long the document is. Higher score, more relevant.

Scoring every document for every term would be slow. Sana stores postings in blocks tagged with each block's best possible score, so once it has a strong top-k it can skip whole blocks that could not beat the current threshold. This rank-safe skipping is called MAXSCORE. 

query: "darkness"

term "darkness"
  block 0  best possible = 3.1  ->  score these postings
  block 1  best possible = 0.2  ->  skip: cannot enter the top-k

Combining searches: hybrid queries

The three questions are strongest together. A multi-query runs several rankings over the one snapshot — say a vector ranking and a BM25 ranking — so their results share a single instant in time and can be fused into one list afterward.

Hybrid retrieval

Blending results from different search types. A common recipe is Reciprocal Rank Fusion: each result is scored by its rank in each list, and the scores are added, so documents that do well in several rankings rise to the top — no need to make incompatible scores comparable. 

one snapshot
  |-- vector top 10   (books that mean like "Dune")
  |-- BM25 top 10     (books mentioning "desert")
  |-- filter          (rating >= 4.5)

fuse the ranked lists  ->  one final ordering, one consistent time
What we earned. A query model that is correct by construction: a frozen snapshot, indexes that may over-answer but are always rechecked, exact filters and honest aggregates, BM25 relevance for text, and a way to combine all of them. Two of the three questions are fully answered. The third — search by meaning — needs vectors.
