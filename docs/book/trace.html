appendix / capstone

One query, end to end.

Every chapter added one layer. Here they run together. We follow a single realistic search through the whole engine, from the pointer in the object store to the ranked list that comes back — naming the chapter behind each step.

The setup

Our library is indexed, but a moment ago someone re-rated The Hobbit (id 5). That write is durable in the WAL but not yet folded into the indexes — so it lives only in the overlay. Now a reader asks a hybrid question: "fantasy books most similar in meaning to this embedding, rated 4.5 or higher, top 3." It touches storage, the overlay, filters, and the full vector pipeline at once. 

recent write (not yet indexed):  id 5 rating 4.5 -> 4.9

query:
  filter      genre = "fantasy" AND rating >= 4.5
  approx_vector  embedding ~ [0.91, 0.04, ...]  k = 3

The walk

  1. Fix a snapshot. Read manifest/current, then the manifest body it names, then the committed WAL cursor. This freezes one consistent view for the whole query. (Ch. 2, Ch. 4)
  2. Build base + overlay. The SST runs and vector files named by the manifest are the base; the WAL slice after the indexed cursor is the overlay. The re-rating of id 5 is in that overlay, so this query will see 4.9, not 4.5. (Ch. 3)
  3. Plan the query. It is a filtered vector search, so Sana takes the materialize path: narrow by filter, search vectors, recheck, rank. (Ch. 4)
  4. Narrow by filter. The attribute index returns a superset of ids with genre = "fantasy" — generous on purpose, to be rechecked later. (Ch. 4)
  5. Probe the IVF. Compare the query embedding to the centroids, pick the nearest few clusters, and scan only those — skipping clusters whose filter bitmap holds no fantasy row at all. (Ch. 5)
  6. Estimate with RaBitQ, then prune. Use the one-bit codes to estimate each candidate's distance; drop any whose best-possible distance cannot reach the top 3. (Ch. 6)
  7. Check liveness. Consult versions.bin and discard stale vector copies, keeping one live vector per id. (Ch. 5)
  8. Exact rerank. Score the few survivors with full precision, and score the overlay's fresh vectors exactly too, then merge — so a just-written vector ranks correctly against indexed ones. (Ch. 5)
  9. Recheck the filter. For each surviving candidate, load the live document and confirm it really is fantasy and really is rated ≥ 4.5. Id 5 passes because the overlay lifted it to 4.9. (Ch. 4)
  10. Rank, limit, return. Order by score, keep the top 3, and return each document with its score. (Ch. 4)

What actually got read

Notice how little of the store was touched. The query followed one pointer, read one catalog, scanned a couple of clusters, and read a handful of documents to recheck. Most of those objects are immutable, so on a warm cache they never reached S3 at all — the difference the latency demo made visible. 

manifest/current          1 small read   (mutable, not cached)
manifest/g/41.json        1 read         (immutable, cacheable)
wal overlay               a few reads    (the fresh tail)
attribute postings        filter superset
ivf base + deltas         only probed clusters
doc SSTs                  one block per rechecked id
The whole book, in one request. Durable storage and a CAS pointer (Ch. 2); sorted runs plus a live overlay (Ch. 3); a snapshot with a superset-and-recheck filter (Ch. 4); IVF approximate search with a version map (Ch. 5); RaBitQ estimate-then-rerank and immutable deltas (Ch. 6); served and cached by a role-split process (Ch. 7). No layer is wasted, and none is trusted blindly.
