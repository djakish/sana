03 / storage shape

Sorted files that are never edited.

Chapter 2 said every data file is immutable — written once, never changed. That sounds incompatible with a database, where data is updated constantly. This chapter shows the shape that makes the two fit: sorted files, stacked newest-first, and quietly merged in the background.

The tension

We want two things at once. Reads should be fast: finding book id 3 should not mean scanning everything. And writes must stay immutable: we cannot edit an old file in place. A single growing, sorted file would give fast reads but would have to be rewritten on every change — exactly the editing we banned. The answer is to keep many small sorted files and never touch an old one. We build that up in two steps: first the file format, then the stack of files.

The SSTable

The file format is an SSTable — a "sorted string table." It is a list of key-value pairs sorted by key and frozen. For our library the key is the book id and the value is the whole document.

SSTable (SST)

An immutable file of key-value pairs sorted by key. Because it is sorted, you can find any key with a few jumps instead of a linear scan; because it is immutable, it can be cached and shared without fear of staleness.

Reading the whole file just to fetch one key would waste the sorting. So an SST is split into blocks of consecutive keys, and a small index at the end records the last key in each block and where it sits. A footer points at the index and carries a checksum so corruption is caught. To find a key, Sana reads the footer, then the index, then exactly one block — three small ranged reads, no matter how large the file. 

[ data block 0 | crc ]   keys 00..19
[ data block 1 | crc ]   keys 20..39
[ data block 2 | crc ]   keys 40..59
[ data block 3 | crc ]   keys 60..79
[ index: last key of each block -> offset, size ]
[ footer: where the index is, checksum, magic ]

A stack of runs

Now the second step. Each time Sana folds a batch of WAL writes into an SST, it creates a brand-new file — a run — and leaves the older runs untouched. Updates and deletes do not edit anything; they are simply newer records in a newer run. This stack of runs, oldest at the bottom and newest on top, is a log-structured merge tree, or LSM.

Run

One sorted SST produced by a single flush. Newer runs sit above older ones. To read a key, Sana checks runs newest-first and stops at the first one that contains it — so the latest version always wins.

How is a delete recorded without editing the old run that still holds the document? With a tombstone: a small marker in a newer run that says "this id is gone." Because newer wins, the tombstone hides the old document on read, even though the old bytes are still sitting there.

The cost of never editing is that runs pile up, and a read may have to check many of them. So a background job called compaction merges runs together: it keeps only the newest record for each id, drops tombstones whose victims no longer exist below them, and writes one tidy run in place of many. Reads stay cheap; the old files become orphans and are reclaimed later.

The freshest writes: the overlay

One gap remains. A write is durable the instant it hits the WAL (Chapter 2), but indexing into an SST happens a little later. So at any moment the newest writes live only in the WAL, not yet in any run. If queries read only the runs, they would miss those writes.

Sana closes the gap with the overlay. The manifest records how far the indexes have absorbed the WAL (the indexed_cursor); the commit state records how far writes are durable. The slice of WAL between them is read live and applied on top of the runs at query time.

Overlay

The not-yet-indexed tail of the WAL — the range from the indexed cursor up to the latest committed cursor. A query reads the SST runs as its base and replays this short overlay on top, so it always reflects the very latest write. This is what gives Sana read-after-write consistency. 

indexed_cursor = 0:104     (runs contain everything up to here)
committed      = 0:109     (writes are durable up to here)

base     = the SST runs named by the manifest
overlay  = wal/0/105 .. wal/0/109     (5 fresh writes)
snapshot = base, with the overlay applied newest-wins

The one rule

Every read obeys a single rule: newest wins. For a full scan, Sana merges the runs in order and then applies the overlay. For a single-key lookup, it skips any run whose stored id range cannot contain the key, then reads just the footer, index, and one block of the rest.

The LSM is not a cache that can be thrown away — it is the indexed form of the durable log. The WAL remains the authority for any write that has not yet become a run. Lose the runs and they can be rebuilt; lose the WAL and you have lost data.
What we earned. A way to store ever-changing data in never-changing files: sorted SSTs for fast lookups, a newest-first stack with tombstones for updates and deletes, compaction to keep reads bounded, and an overlay so the latest write is always visible. We can now answer real questions over this shape.
