Running it for real.

A design is not finished until it says who is allowed to do work, how a process starts and stops without dropping requests, and — honestly — what is still missing. This chapter is about turning the engine into something you can run and watch.

Who does the work: roles

Three kinds of work happen in Sana: serving reads and writes, turning the WAL into indexes, and background upkeep (compaction and vector maintenance). For development, one process does all three — sana serve runs the HTTP server with an indexing worker and a maintenance loop built in, so your writes become searchable with no second process to manage.

For a real deployment you split them into roles, each scaled on its own. The reason is concrete: if every query-serving process also scanned namespaces and claimed indexing jobs, then adding capacity for reads would multiply background traffic to the object store. Separating roles keeps query scaling from dragging upkeep along with it. 

serve --role all          one process: http + indexing + maintenance (dev)

serve-api                 http only — never touches background work
work-indexing --loop      claim a namespace, fold its WAL, publish
maintain --loop           compaction and vector maintenance across namespaces

Coordinating safely: the queue

When a write commits, how does an indexing worker learn there is work to do? Each commit drops a best-effort note into a shared queue — a single JSON object in the store — naming the namespace that changed. Workers claim namespaces from it under a fenced, time-limited lease, so exactly one worker indexes a namespace at a time.

The subtle part is that the queue is only a hint. The WAL and manifest remain the authority. If a notification is ever lost, a reconciliation pass compares the committed cursor against the indexed cursor, notices a namespace fell behind, and re-enqueues it. Nothing depends on the queue being perfect. 

write commits cursor 0:77
  |
  +-- best-effort: enqueue "books"

if that note is lost:
  reconcile sees committed 0:77 > indexed 0:70  ->  enqueue "books" again

Because it is one object updated by compare-and-set, heavy write traffic means heavy contention on that single key — a known scaling limit, and the reason the repository still tracks a future standalone queue broker.

Starting and stopping without dropping requests

A process that is alive is not always ready to serve. Sana exposes both facts separately, which is exactly what an orchestrator like Kubernetes needs.

Liveness vs. readiness

Liveness says the process is running at all — if it fails, restart the process. Readiness says the process should receive traffic right now — if it fails, stop routing requests here, but do not kill it. A process can be alive yet deliberately unready.

That distinction makes a clean shutdown possible. On Ctrl-C or a termination signal, Sana first flips readiness off and rejects new namespace traffic, waits a moment for load balancers to notice, and only then lets in-flight HTTP requests drain to completion before exiting. Indexing and maintenance loops check for the same signal between jobs, so they never start new work mid-shutdown.

EndpointWhat it answers
/livez, /healthzIs the process alive? (does not depend on S3)
/readyzShould it get traffic? Fails during startup, drain, overload, or backend trouble.

Where the milliseconds go

Recall the object store's one weakness: every request to it costs tens of milliseconds. Sana's defense is a cache that holds immutable objects in the process. Because those objects can never change (Chapter 2), caching them is always safe; only mutable pointers and cursors skip the cache and go straight to the store. The request meter sits below the cache, so a cache hit is not counted as backend traffic. That keeps the /metrics numbers honest: they measure true round trips to S3, not work the cache absorbed. The same layering gives latency histograms split by phase and gauges for cache temperature and per-namespace indexing lag.

What is honestly still unfinished

A teaching project earns trust by naming its edges. Several hard problems are deliberately left fail-closed rather than done halfway. The biggest is safe automatic deletion across many machines: reclaiming a file is only safe once you can prove no reader and no in-flight publisher still needs it, and Sana now records active query/recall reader snapshots but still lacks publisher safety points and durable GC candidate state. So automatic garbage collection is off by default; deletion stays an operator action. 

safe deletion needs proof:
  no reader still holds the old generation
  no publisher can still attach the object
  the candidate survived a durable safety check
  the final reference check still passes immediately before delete

Also not present: authentication and TLS on the HTTP service (bind to localhost or front it with a reverse proxy), WAL epoch rotation, IAM-role S3 credentials, publisher-safe automatic GC, and a turbopuffer-compatible wire protocol. None of these are hidden; they are tracked in the repository.

What we earned. A system that can be split by role, scaled, monitored, and shut down without dropping a request — plus an honest map of its frontier. To see all the parts work together, the appendix follows one query through every layer of the book.
