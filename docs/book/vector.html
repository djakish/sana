05 / retrieval

Searching by meaning.

Filters match facts and full-text matches words. But "find me books that feel like Dune" is neither — it is about meaning. Answering it is what vectors are for. We start with what a vector even is and build up to searching millions of them.

What a vector is

A vector is just an ordered list of numbers. That's all. [0.9, 0.1] is a vector with two numbers; [0.2, 0.8, 0.5] has three. The useful way to picture it is as a point in space: two numbers place a point on a flat map (an x and a y), three numbers place it in a room, and although we can't picture it, a list of 768 numbers places a point in 768-dimensional space the same way.

Vector

An ordered list of numbers, understood as the coordinates of a point. The count of numbers is the number of dimensions. Two points being "close" in this space is the entire basis of vector search.

Where meaning comes from: embeddings

A list of numbers has nothing to do with books or meaning — until we choose the numbers carefully. An embedding is a vector produced by a trained model whose whole job is to place similar things near each other. Feed it the text of Dune and it returns a few hundred numbers; feed it The Left Hand of Darkness and it returns another few hundred — and because both are dense, thoughtful science fiction, those two points land near each other, while a cookbook lands far away.

Embedding

A vector that represents the meaning of something — a sentence, a document, an image — created by a model trained so that similar inputs map to nearby points. The numbers individually mean nothing; only their positions relative to other embeddings carry the meaning.

You do not need to understand how the model is trained to use it. For this book, an embedding is a black box: text in, a point in space out, with the one guarantee that closeness equals similarity. Sana stores these points; it does not produce them.

Closeness is similarity

If similar meanings sit at nearby points, then "find similar books" becomes "find nearby points." To search, we embed the query into the same space and look for the stored points closest to it. Drag the query below: notice that what counts is the direction a point sits in — items pointing the same way are judged most alike.

The honest way: check everything

The simplest correct search is exact k-nearest-neighbor search: compare the query to every stored vector, score each with the metric, sort, and return the best k. For our five books this is instant and perfectly accurate.

k-nearest-neighbor (kNN)

Returning the k stored items closest to the query. Exact kNN compares against every item, so it is always right — and its cost grows with the size of the collection. 

query q = [1.0, 0.0]

id 3  [0.91, 0.04]  ->  score
id 2  [0.20, 0.80]  ->  score
id 4  [0.70, 0.20]  ->  score
...every vector...

sort by score, return top k

It is also a trap at scale. Ten million embeddings of 768 numbers each is billions of multiplications per query. Exact search reads and scores the entire collection every time. We need a way to look at only a small, promising slice.

The bargain: approximate search

Approximate nearest neighbor search trades a sliver of accuracy for an enormous speed-up. The idea is to inspect only the regions of space where good answers are likely to live, ignore the rest, and then score exactly the few survivors. The approximation decides what to look at; an exact rerank decides the final order, so the results that come back are still scored truthfully.

Recall

The fraction of the true nearest neighbors that an approximate search actually returned. Recall of 0.95 means it found 95 of the true top 100. Vector search is the art of pushing recall close to 1.0 while reading as little as possible — and you can dial the trade-off per query.

IVF: divide the space into cells

Sana's approximate index is the inverted file, or IVF. Build it in two moves. First, run k-means, an algorithm that finds a set of representative points called centroids — think of them as the "capitals" of natural clusters in the data. Second, file every vector under its nearest centroid, so the space is carved into cells.

Centroid & cluster

A centroid is the center of a group of nearby vectors; the group filed under it is a cluster (or partition). At query time Sana compares the query only to the centroids, picks the nearest few, and scans just those clusters. The number it scans is the probes setting — your accuracy-versus-speed dial.

To search, compare the query to the handful of centroids, probe the nearest clusters, scan only the vectors filed there, and exact-rerank the survivors. Slide the probe count and watch the candidate set — and the cost — grow as you trade speed for recall:

IVF probes

One probe is fast but might miss a near point that sits just across a cell border; more probes catch it at more cost. The query marked q is reranked exactly against the vectors in the lit clusters only.

Filtered vector search

Often the meaning question carries a fact alongside it: "books like Dune, but only fantasy." A naive engine would find the nearest vectors and then throw most away for failing the filter — wasted work, and it can return too few. Sana instead stores the filter information beside the IVF data as bitmaps: a cluster-level summary says whether a cluster holds any allowed row at all, and a row-level bitmap says which vectors inside it are eligible. Disallowed clusters are skipped whole. 

filter: genre = "fantasy"

cluster 0: contains a match  ->  scan rows allowed by mask 101001
cluster 1: no match at all   ->  skip the entire cluster
cluster 2: contains a match  ->  scan rows allowed by mask 001110

Updates without editing: the version map

Vector files are immutable like everything else (Chapter 2), so an update cannot edit a stored vector. Instead it writes a new copy in a new segment, and a small version map records which copy is the live one for each id. The old vector still sits on disk, but ranking ignores it because the map does not point at it.

Version map (liveness)

A file (versions.bin) mapping each document id to the version of its currently-live vector. A vector copy counts only if the map points at it; stale copies are invisible without ever being deleted. This is how immutable vector files support live updates. 

versions.bin:  id 17 -> version 4

segment A:  id 17, version 3   ->  stale, ignored
segment B:  id 17, version 4   ->  live, ranked
What we earned. Search by meaning: documents as embedding points, closeness as similarity, exact kNN for correctness and IVF for speed, filters fused into the scan, and a version map so updates work on immutable files. The last chapter on vectors makes this fast index stay fresh and cheap as data changes.
