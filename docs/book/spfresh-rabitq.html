06 / freshness and quantization

Staying fresh, staying cheap.

The IVF index from Chapter 5 has two weaknesses once data keeps changing. Its clusters drift out of shape, and scoring full-precision vectors is still heavy. Two ideas fix them: SPFresh keeps the clusters healthy locally, and RaBitQ makes each comparison almost free.

Why a cluster index rots

The centroids were chosen by looking at the data as it was when the index was built. As new books are added and old ones change, some clusters swell huge and others empty out. A bloated cluster means scanning thousands of vectors on a probe; a lopsided layout means the nearest centroid is no longer a good guide to the nearest vector. Recall quietly drops.

There are two obvious responses, and both are bad. Rebuild the entire index from scratch on every change — correct, but ruinously expensive at scale. Or never rebuild — cheap, but the index decays until it is useless. We want the middle: fix only the parts that actually went out of shape.

SPFresh (LIRE)

A design for keeping a cluster index healthy with local repairs instead of global rebuilds. Its rule, LIRE, does three small things: split a cluster that has grown too big, merge one that has shrunk too small, and reassign the handful of boundary vectors near the clusters it just changed. The rest of the index is left untouched. 

base IVF
  c0: 1200 rows
  c1:  980 rows
  c2:  210 rows   (too small)
  c3: 1810 rows   (too big)

local plan
  c3 -> split into two
  c2 -> merge with its nearest neighbor
  boundary vectors nearby -> reassign to the right cluster

Sana's twist: immutable deltas

SPFresh as published edits the index in place. Sana cannot — every file is immutable (Chapter 2). So instead of mutating the base, Sana publishes the repair as a new file: a small delta segment for each append, reassignment, or local rebuild. The version map from Chapter 5 then makes the newest copy of each vector the live one. 

vector family for "embedding"
  ivf.bin                  base index
  append-42.ivf.bin        new vectors since the base
  reassign-43.ivf.bin      boundary vectors moved
  local-rebuild-44.ivf.bin a cluster rebuilt locally
  versions.bin             which copy is live per id

query: scan base + deltas, drop stale versions, rerank the live ones

This is the same pattern as the LSM in Chapter 3: never edit, always append a newer layer, and let a liveness rule decide what counts. Compaction later folds the deltas back into a clean base, exactly as it merges SST runs.

RaBitQ: shrink every vector to bits

The second weakness is raw cost. Even after IVF narrows the field, exact- scoring each survivor means crunching hundreds of full-precision numbers. Quantization is the fix: replace each number with a much smaller approximation, so a first-pass comparison is cheap.

Quantization

Compressing numbers by storing them at lower precision — here, all the way down to a single bit per dimension. The compressed form is too rough for a final answer but plenty good for quickly deciding which vectors are worth a full, exact comparison.

RaBitQ does this cleverly. For a vector o in a cluster with centroid c, it works with the residual — the offset from the centroid, o − c — because within a cluster the residuals are what tell vectors apart. It normalizes that residual to unit length, rotates it by a fixed random transform (which spreads information evenly across the bits), and keeps just the sign of each dimension: one bit. Alongside the bits it stores the residual's length and a small correction term so the rough estimate can be de-biased. 

vector o, centroid c

residual       r  = o - c
unit residual  u  = r / length(r)
rotated        u' = P * u            (fixed random rotation)
code bits      b  = sign(u')         (one bit per dimension)
stored         length(r), correction

Estimate, prune, then rerank

At query time Sana rotates the query's residual once per cluster, then uses fast bit operations to estimate the distance to every code. Crucially, RaBitQ comes with a mathematical error bound, so each estimate has a known best-case. If a candidate's best possible distance still can't beat the current top-k, it is pruned without ever touching its full vector. Only the genuine contenders are scored exactly. 

for each probed cluster
  rotate the query residual
  estimate distance from the bits        (cheap)
  compute the lower bound
  if the lower bound cannot win -> prune  (skip the full vector)
  else -> exact distance against the full vector
RaBitQ is never the final answer; it is a cheap test that decides which expensive comparisons are still worth paying for. The exact rerank — the same one from Chapter 5 — always has the last word, so quantization speeds things up without making the ranking wrong.

Why the pieces fit

IVF gives clusters. SPFresh-style maintenance keeps those clusters the right size and shape as data churns. RaBitQ reuses the very same clusters to store a one-bit sketch per vector. And the object-store rules from Part II — immutable files, a version map, catalog-last publication — let all of it be appended and published without ever editing anything.

What we earned. A vector index that stays accurate while data changes and stays cheap while it is queried — both achieved by appending immutable deltas rather than editing, exactly like the rest of Sana. The engine is complete; what remains is running it.
